# MVP 功能範圍定義

> **目標**：定義最小可行產品範圍，聚焦解決「搜尋的人多，收藏的人少」
> **原則**：先讓核心體驗做好，再擴展功能

---

## 一、MVP 核心假設

### 要驗證的假設

```
假設 1：正餐決策是最大痛點
驗證方式：正餐時段使用率、隨機功能使用率

假設 2：情境標籤（聚餐/獨食/飲控）有價值
驗證方式：情境篩選使用率、滿意度

假設 3：「隨機推薦」可以解決選擇困難
驗證方式：隨機功能使用率、重複使用率

假設 4：收藏功能配合地圖視覺化有價值
驗證方式：收藏數、收藏後查看率

假設 5：比 Google Maps 更好的餐廳探索體驗
驗證方式：用戶留存率、NPS

假設 6：社會證明（XX 人收藏）增加信任
驗證方式：有社會證明 vs 無社會證明的收藏轉換率
```

### MVP 成功標準

```
數量指標：
├─ 100 個註冊用戶
├─ 總收藏數 1,000+
├─ 每用戶平均收藏 10+ 家
└─ 7 日回訪率 > 25%

質量指標：
├─ 用戶反饋正面
├─ NPS > 30
└─ 有自發分享行為
```

## 三、功能優先級排序

### P0：沒有就不能上線

| 功能 | 說明 | 驗證假設 |
|------|------|---------|
| 社群登入 | Google 一鍵註冊 | 降低註冊摩擦 |

| OG Image 預覽 | 分享連結預覽圖 | 提升分享點擊率 |

| 功能 | 說明 | 驗證假設 |
|------|------|---------|

| 功能 | 說明 | 驗證假設 |
|------|------|---------|
| 附近推薦 | 位置推薦 | 增加價值 |
| 收藏統計 | 收藏里程碑 | 成就感 |

## 六、不做清單（明確排除）

### 這些功能 MVP 不做

| 功能 | 排除原因 |
|------|---------|
| 用戶評論 | 需要審核機制、內容品質管理 |
| 用戶評分 | 與 Google 評分重複 |
| 社群功能 | 需要用戶基數 |
| AI 推薦 | 需要數據累積 |
| 通知系統 | 先驗證核心功能 |
| 手機 App | Web 優先 |

### 這些優化 MVP 不做

| 優化 | 排除原因 |
|------|---------|
| 無限滾動 | 分頁足夠 |
| 離線支援 | 非必要 |
| 多語言 | 先做繁體中文 |
| PWA | Web 優先 |

## 七、開發順序建議

### Sprint 1（Week 1-2）：核心體驗優化

```
目標：讓用戶可以順暢地探索和收藏餐廳

任務：
□ 首頁隨機推薦功能突顯
□ 收藏按鈕 UX 優化
□ 一鍵收藏流程
□ 空狀態設計實作
□ 文案調整
□ 收藏人數顯示（「XX 人收藏」）
□ OG Image 分享預覽優化
```

### Sprint 2（Week 3-4）：收藏價值展示

```
目標：讓用戶感受到收藏的價值

任務：
□ 收藏地圖視覺化
□ 收藏統計顯示
□ 收藏頁面優化
□ 去過標記功能優化
□ 收藏里程碑（簡易版）
```

### Sprint 3（Week 5-6）：探索體驗優化

```
目標：讓用戶更容易發現好餐廳

任務：
□ 篩選器 UX 優化
□ 搜尋功能優化
□ 地圖聚類優化
□ 附近推薦功能
```

### Sprint 4（Week 7-8）：優化與上線

```
目標：準備上線

任務：
□ 分享預覽優化
□ 效能優化
□ Bug 修復
□ 種子用戶招募
□ 上線
```

## 九、風險與應對

### 風險 1：開發時間超出預期

```
應對：
├─ 嚴格遵守「不做清單」
├─ 如果來不及，先砍「MAYBE」功能
└─ MVP 不求完美，求驗證
```

### 風險 2：上線後沒人用

```
應對：
├─ 先準備種子用戶（冷啟動手冊）
├─ 不要「Build it and they will come」
└─ 上線前就要有 30 個種子用戶準備好
```

### 風險 3：假設錯誤

```
應對：
├─ 快速上線、快速學習
├─ 準備好 Pivot 的心態
└─ 用戶訪談比數據更重要
```
